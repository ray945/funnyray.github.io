AWS IoT KeepAlive 心跳流程

发表于 2019-06-15 更新于 2020-03-23 分类于 经验总结

首先简单说一下，AWS Iot SDK 的 mqtt 模块是基于 paho 标准的 mqtt client java 进行开发的，没有加入 paho 的 android 扩展包，所以心跳只能通过 TimerPingSender 进行定时发送。但由于在 android 系统中，如果系统长时间没有 cpu 活动，会进入休眠状态，而简单的 Timer 是无法唤醒 cpu 的，这就导致官方的 AWS IoT SDK 无法理想地运行在游乐园项目（相机需一直保持 mqtt 在线）。所以游乐园项目使用的 AWS IoT SDK 是我们手动加入了android 扩展包中的 AlarmPingSender 后自己编译的。

AlarmPingSender

AlarmPingSender 与 原生的 TimerPingSender 大同小异，都继承自 MqttPingSender，并实现 init()，start()，stop()，schedule(long delayInMilliseconds) 四个方法。区别在于：schedule() 方法中，TimerPingSender 使用 Timer 设置定时来发送下一次心跳，而 AlarmPingSender 使用 AlarmManager 设置定时。这里介绍一下 AlarmPingSender 中的这四个方法以及一个广播接收器。

init()

init() 方法里是简单的初始化操作，传入一个 ClientComms 对象，之后创建一个广播接收器 AlarmReceiver

start()

start() 方法，首先通过 clientId 拼接一个专属的 action，并把该 action 注册到之前创建的 AlarmReceiver 中，然后通过 PendingIntent 发送第一次心跳广播（注意，这里只是发送广播，具体心跳的发送会在 AlarmReceiver 中介绍），接着通过调用 schedule() 方法设置下一次心跳广播的定时，间隔为一次心跳时间，最后把标志位 hasStarted 置为 true

stop()

stop() 方法与 start() 相反，取消正在发送的心跳广播，把标志位 hasStarted 置为 false，注销广播接收器 AlarmReceiver

schedule(long delayInMilliseconds)

除了第一次心跳广播是通过 start() 发送的以外，之后的每一次心跳广播都是通过 schedule() 发送

AlarmReceiver

AlarmReceiver 才是整个 PingSender 发送心跳的核心业务部分。每一次心跳广播都由 AlarmReceiver 接收，收到广播后会持有 wakelock，然后通过 ClientComms.checkForActivity() 发送心跳。该方法在 mqtt client 基础 sdk 中，这里就不细讲了，主要就是向 mqtt 服务器发送心跳包。
